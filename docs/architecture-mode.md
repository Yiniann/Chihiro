# 架构模式选择

这份文档用于确认 Chihiro 当前阶段的产品模式和渲染策略。

当前决定已经明确：

1. Chihiro 第一阶段做单租户
2. Chihiro 不是纯静态博客，而是单站点 CMS
3. 公开内容优先走静态化能力，后台保持动态

## 当前目标

从目前的需求来看，Chihiro 的目标是：

- 有一个公开前台
- 有一个控制中台
- 文章可管理
- 站点描述、SEO 配置等信息后续可在后台修改
- 支持 RSS、SEO、sitemap

这意味着 Chihiro 当前更适合被定义为：

**单站点博客 CMS**

而不是：

- 单纯的静态博客模板
- 多租户博客平台

## 为什么不做多租户

当前阶段不做多租户，原因很直接：

- 会显著提高数据库和业务复杂度
- 会增加路由、缓存、域名和配置管理成本
- 会分散前台页面、后台管理、SEO 和内容系统的开发精力

对于现在这个项目，更重要的是先把下面这些能力做完整：

- 前台体验
- 后台内容管理
- SEO
- RSS
- 发布流程

## 当前采用的模式

## 产品模式

采用：

**单租户、单站点 CMS**

也就是：

- 只有一个公开站点
- 只有一套站点设置
- 只有一套后台管理
- 不引入租户隔离、租户域名、租户主题等平台级能力

## 渲染模式

采用：

**公开内容页静态优先，后台动态渲染**

具体建议如下：

- 首页：ISR
- 文章列表页：ISR
- 文章详情页：ISR 或静态参数生成
- RSS：动态生成或按需更新
- sitemap：动态生成或按需更新
- admin：动态渲染
- 登录相关：动态渲染

## 为什么不是纯 SSG

虽然是单租户，但当前仍然不建议把整个系统定成“全量纯 SSG”。

原因是：

- 后台存在动态数据和登录态
- 站点配置后续希望可以改
- 发布文章后希望页面可以较快刷新

所以更稳妥的方案是：

- 公开内容：SSG / ISR 思路
- 后台能力：动态

## 为什么不是纯 SSR

也不建议把公开内容全部做成纯 SSR。

因为博客最核心的诉求仍然是：

- SEO
- 页面速度
- 稳定首屏
- RSS / sitemap / metadata 友好

这些都更适合静态优先的公开页面。

## 项目形态建议

当前阶段建议继续保持：

- 一个 Next.js 项目
- 前台、admin、api 共存

也就是继续使用现在这类结构：

```text
src/app
  page.tsx
  posts/
  admin/
  api/
```

这样开发成本最低，也最适合第一阶段快速推进。

## 数据模型建议

既然当前确定做单租户，那第一阶段不需要额外引入 `tenantId`。

建议优先定义这些核心模型：

- `siteSettings`
- `users`
- `posts`
- `tags`
- `categories`

如果后续真的转向平台化，再评估是否补租户层。

## 当前确认结论

当前可以明确写成一句话：

**Chihiro 第一阶段按单租户单站点 CMS 开发，公开内容页采用 ISR，后台与管理页采用动态渲染。**

## 这个决定带来的直接影响

一旦确认走单租户，接下来很多事情都会简单很多：

- 不需要设计租户隔离
- 不需要设计租户域名路由
- 不需要为文章和站点配置加租户归属
- 不需要为缓存和 revalidate 先考虑平台级问题

我们就可以把注意力集中在：

- 内容模型
- 页面结构
- 后台管理
- SEO / RSS
