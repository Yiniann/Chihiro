# Chihiro Site 主题设计系统

这份文档用于定义 Chihiro 前台站点的视觉语言、组件规则和页面气质。

它服务的不是某一个页面，而是公开 site 的整体主题，包括：

- 首页与导航
- 内容阅读页
- `/more` 与各独立栏目页
- `/projects` 等展示型页面
- 列表、卡片、表单、空状态、导航等共享模块

## 一句话原则

Chihiro 的前台主题应该像一个被长期打磨的站点，而不是一组拼出来的页面。

## 1. 设计目标

这套主题要同时满足四件事：

- 有清晰稳定的品牌感
- 适合长内容阅读
- 适合公开展示与浏览
- 在明暗主题下都保持统一气质

成功标准：

- 新页面即使没有专门设计稿，也能从现有 token 和组件规则自然长出来
- 首页、内容页、列表页和展示页虽然用途不同，但能看出属于同一个系统
- 用户首先感受到的是内容与层级，而不是视觉噱头

## 2. 主题定位

Chihiro 的前台主题不应被理解为“博客模板风格”，而应被理解为：

- 一套内容产品主题
- 一套兼顾阅读、浏览和展示的界面语言
- 一套轻品牌感、轻产品感、轻氛围感的系统

整体气质关键词：

- 克制
- 清洁
- 精致
- 有呼吸感
- 带一点冷静的数字感

不追求：

- 强装饰
- 强拟物
- 网页模板感
- 过度科技感

## 3. 设计系统边界

这份设计系统覆盖三层内容：

- Foundation：颜色、字体、间距、圆角、阴影、动效
- Component：按钮、输入框、卡片、导航、标签、空状态
- Pattern：首页、内容页、列表页、栏目页、展示页

以后如果新增页面，应优先复用这三层，而不是为单页单独找风格。

## 4. Foundation

### 4.1 颜色系统

当前主题已经有明确的基础色结构，核心来自 `src/app/globals.css`：

- `--primary`
- `--background`
- `--foreground`
- `--site-canvas-background`
- `--site-shell-background`

颜色角色应固定如下：

- `primary`：交互强调、选中状态、重点标题、链接和少量高亮
- `background`：页面基础背景
- `foreground`：正文与主体文本
- `zinc` 中性色：边框、次级文本、弱背景、结构层次

规则：

- 一个主题只有一个主强调色
- 大部分页面层次靠中性色完成
- 强调色只在真正需要“注意我”的地方出现
- 不允许为不同页面随意换主色

### 4.2 明暗主题

Chihiro 不是“做完亮色后补一个暗色”，而是双主题并列设计。

规则：

- light 和 dark 共享同一套语义 token
- 变化的是数值，不变化的是角色
- 暗色模式不应只是“全部反色”
- 暗色模式要控制对比度，避免刺眼和脏灰

当前系统特征：

- light 主题偏柔和灰白底
- dark 主题偏深墨底 + 冷色主强调
- 背景不是纯平色，而是带弱 radial glow 的空间感画布

### 4.3 背景系统

背景是 Chihiro 主题辨识度的重要组成部分。

当前已有两个关键背景 token：

- `--site-canvas-background`
- `--site-shell-background`

规则：

- 页面大背景允许有非常轻的氛围层
- 氛围层只能做空间感，不能影响信息可读性
- 组件表面必须比大背景更稳定、更易读
- 页面背景可以有气氛，但正文承载面必须克制

### 4.4 字体系统

当前全局字体角色已经建立：

- `--font-sans`
- `--font-serif`
- `--font-mono`

即使当前 serif 仍复用接近 sans 的设定，角色也应该先明确。

推荐角色分工：

- Sans：导航、按钮、表单、列表、UI 文本
- Serif：长标题、栏目头、导语、长文重点段落
- Mono：代码、技术元信息、标签、数字、状态辅助信息

规则：

- 不要所有内容都用一种语气说话
- UI 与长文应有微妙但明确的气质区分
- mono 用来表达精确感，不用来装饰

### 4.5 字号层级

字号应优先服务阅读和层级，而不是追求视觉变化。

推荐语义层级：

- Hero：页面主标题
- H1：页面标题
- H2：区块标题
- H3：组件标题
- Body：正文和 UI 基础文本
- Meta：说明、标签、次级信息
- Eyebrow：小帽子文字、分组提示

规则：

- 标题层级靠尺寸、字重、行高共同建立
- 不通过乱用颜色代替层级
- 小字必须保持清晰，不能只为“高级感”而过细

### 4.6 间距系统

留白是 Chihiro 的主设计工具之一。

规则：

- 页面区块间距大于组件内部间距
- 同级元素使用稳定间距，不随页面心情变化
- 卡片内部要有标题组、内容组、元信息组的分层
- 不要靠塞更多元素来显得“丰富”

### 4.7 圆角系统

圆角是主题气质的一部分，当前系统已经明显偏柔和、现代。

推荐约定：

- 小标签：小圆角或胶囊
- 按钮 / 输入框：中到大圆角
- 卡片：`rounded-xl` 到 `rounded-2xl`
- 重点容器：更大的柔和圆角

规则：

- 圆角表达的是界面性格和层级
- 同类组件圆角必须统一
- 不同栏目可以密度不同，但不要变成两套性格

### 4.8 阴影与边框

Chihiro 的层次应该主要来自边框、背景和轻微浮起感，而不是重阴影。

规则：

- 默认优先用边框区分表面
- 阴影只做轻微抬升
- hover 可以比静态稍微更浮起
- 禁止厚重、发黑、攻击性强的投影

一句话约束：

边框是主语，阴影是修饰。

### 4.9 动效系统

动效应该帮助主题显得有生命，而不是显得忙。

当前已有的方向是正确的：

- 主题切换有 aura 扩散感
- 过渡使用平滑的 cubic-bezier
- backdrop blur 与透明层次配合使用

规则：

- 动效只服务状态变化和层级感知
- 不要给普通内容加表演型动画
- hover 动效时间要短，切换动效可以稍长
- 尊重 `prefers-reduced-motion`

## 5. Component 规范

### 5.1 按钮

按钮需要明确层级，至少区分：

- Primary
- Secondary
- Ghost

规则：

- primary 只用于最主要操作
- secondary 用于同层次次要操作
- ghost 用于低干扰辅助动作

视觉上：

- 高度稳定
- 圆角统一
- hover 主要通过边框、亮度和轻阴影变化表达

### 5.2 输入框与表单

表单要像站点的一部分，而不是浏览器默认控件拼接。

规则：

- 输入框表面清晰、边界明确
- focus 态优先通过 `primary` ring 和边框表现
- placeholder 只做提示，不能比正文更抢眼
- 订阅、搜索、评论、友链申请等表单都要说同一种视觉语言

### 5.3 卡片

卡片是整个主题里使用频率最高的表面组件之一。

规则：

- 卡片必须有明确内容层次
- 列表容器退后，单张卡片承担结构重点
- 卡片 hover 只做轻微抬升和边框变化
- 透明、模糊、弱高光可以用，但必须克制

适用范围：

- 项目卡片
- 推荐内容卡片
- 占位内容块
- 栏目引导面板

### 5.4 标签与徽标

标签用于压缩信息，不用于堆花样。

规则：

- 标签文案尽量短
- 状态标签和 topic 标签必须区分语义
- badge 可以用主色弱底，但不能把页面切成彩色碎片

### 5.5 导航

导航是系统感最强的组件之一。

- 轻
- 稳
- 不压内容

站点导航规则：

- 默认状态轻量
- 当前栏目反馈清晰
- 滚动后可以增强表面感，但不能压住正文
- 桌面与移动端应保持同一语气，不同交互密度

MegaNav 规则：

- MegaNav 顶部的 section label 如 `Home`、`Posts`、`Updates`、`Timeline`、`More` 属于“面板分组提示”，视觉上必须比面板内部卡片的 eyebrow 更轻、更退后
- 面板内部卡片的英文 eyebrow 如 `Message`、`Projects`、`Friends`、`Bookmarks` 属于“卡片级提示”，可以比 section label 稍大一档，但不能压过中文主标题
- 中文标题如 `留言`、`项目`、`友链`、`书签` 才是卡片内的主要信息，层级必须高于英文 eyebrow
- 不要让 MegaNav 顶部 section label 和卡片内部 eyebrow 使用完全相同的视觉权重，否则层级会塌
- `Timeline` 必须复用通用的 `MegaNavSection` 结构，不能单独写一套局部标题规则，否则同一导航里会出现两套眉标尺度

一句话判断：

- 面板级提示 < 卡片级提示 < 卡片中文标题

### 5.6 空状态

空状态要像“系统当前处于这个阶段”，而不是“报错了”。

规则：

- 说明当前状态
- 给出下一步
- 气质保持平静
- 不用夸张插画撑场面

### 5.7 代码与技术信息

代码块、命令、URL、统计值、版本号等技术信息应统一使用 mono 语气。

规则：

- mono 只在需要精确感时使用
- 行内 code 视觉要清晰但不突兀
- 代码块应比正文更稳，不必比正文更吵

## 6. Pattern 规范

### 6.1 首页与总览页

首页与总览页应优先服务理解、进入和建立第一印象。

规则：

- hero 要有明确记忆点
- 主标题和导语清晰
- 动效只能增强气质，不能干扰阅读
- 从 hero 进入下层内容的节奏要顺

### 6.2 内容页

文章、README、长文内容页应强调阅读舒适度。

规则：

- 行高足够
- 段落节奏稳定
- 标题层级鲜明
- 链接和引用有自己的语气

### 6.3 列表页

列表页应先解决“扫读效率”，再谈装饰。

规则：

- 每行或每卡的重点信息固定
- 筛选和搜索只是辅助，不是主角
- 列表容器应退后，条目本身承担信息层级

### 6.4 展示页

展示页包含 `/projects`、专题页、独立页面等“偏陈列型”的站点页面。

规则：

- 展示页不是素材堆砌页
- 重点是叙事、选择和层级
- 指标信息只能辅助，不能喧宾夺主
- 展示组件要比普通列表更有呼吸感

### 6.5 占位页与建设中页面

占位页也要符合主题语言，不能像临时补丁。

当前 `MoreSectionPlaceholder` 的方向是合理的：

- 大标题
- 导语
- 柔和的占位面板
- 明确回退动作

## 7. 内容语气

Chihiro 的文案语气应与界面气质一致。

推荐语气：

- 平静
- 直接
- 有判断
- 不卖弄

避免：

- 过度营销化
- 过度俏皮化
- 大量口号式文案

## 8. 反模式

以下做法应明确避免：

- 每个页面单独找一套风格
- 不同区域使用完全不同的圆角、阴影和边框语言
- 把主色当成大面积装饰色到处铺
- 暗色模式只是把白底改黑底
- 后台看起来像另一套产品
- 为了显得高级而过度使用 blur、glass、gradient
- 所有强调都靠颜色，没有结构层级
- 大量重阴影、重描边、重发光

## 9. 当前系统抽象

基于现有代码，可以把 Chihiro 的主题先抽象成下面这套核心语言：

- 以 `primary` 为唯一主强调色
- 以 `zinc` 中性色承担大部分层次
- 以柔和大圆角建立现代感
- 以轻边框和轻阴影建立表面
- 以前台氛围背景建立站点记忆点
- 以清晰 typography 和留白保持内容中心

## 10. 后续落地建议

如果你后面继续完善整套主题，我建议优先做这几件事：

- 把 token 继续从页面 class 中抽出来，形成更稳定的语义层
- 梳理一组共享 surface 规则，例如 card、panel、input、button
- 明确 serif 的真正使用范围，而不是只保留名义角色
- 统一 site 内各栏目页的表面层级写法
- 给常见 pattern 建一套可复用骨架，而不是每页重新拼

## 11. 验收清单

设计或开发完成后，可以按下面的标准检查：

- 新页面是否一眼能看出属于 Chihiro
- 首页、内容页、栏目页、展示页是否明显属于同一套系统
- light 和 dark 是否只是主题切换，而不是风格切换
- 内容是否始终比装饰更重要
- 是否存在不必要的特效和噪声
- 标题、正文、元信息是否层次明确
- 交互组件是否拥有统一的边框、圆角和反馈逻辑
- 占位页、空状态、栏目页是否也保持系统感

## 12. 最终定义

Chihiro 的 site 主题设计系统不是为了让页面“看起来好看一点”，而是为了让前台的任何新栏目、新页面和新内容进入时，都能自然长成同一种语言。

它要统一的不是装饰，而是判断。
